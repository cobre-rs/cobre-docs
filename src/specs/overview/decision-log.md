# Decision Log

## 1. Purpose

This log is the central registry of cross-cutting architectural decisions for the Cobre
ecosystem. A decision belongs here when it affects two or more spec files — meaning a change
to the decision would require propagated updates across the corpus. Decisions that affect
only a single file remain documented inline in that file and are not cataloged here.

The log is updated in batches: when a decision is made, all affected files are identified
before the entry is written, so the Affected Files column is complete at time of insertion.
When a decision is superseded, the Status column is changed to `superseded` and the new
decision's ID is noted in the Decision Summary; the original entry is never deleted.

This file is the authoritative reference for ticket-013 (inline decision markers) and is
the source of truth for `CLAUDE.md` convention enforcement.

## 3. How to Use This Log

**Adding a new decision.** When an architectural decision is made that affects two or more
spec files, add a row to the registry table in section 2 before (or at the same time as)
propagating the decision to the affected files. Assign the next sequential `DEC-NNN`
identifier. Set Status to `active`. The Affected Files column must list every file that
either documents or depends on the decision -- not just the primary spec. Then place an
inline marker in each affected file (see ticket-013 for the marker format).

**Verifying propagation.** Before closing any PR that touches a decision listed here,
grep each file in the Affected Files column for the relevant term and confirm the decision
is stated consistently. The ID in the inline marker (e.g., `<!-- DEC-001 -->`) enables
automated checking.

**Superseding a decision.** When a decision is replaced, change the old entry's Status to
`superseded` and append "(superseded by DEC-NNN)" to its Decision Summary. Create a new
entry for the replacement. Do not delete the old entry -- the historical record is needed
to understand why files may still reference both the old and new approaches.

**Batch updates.** When modifying this log, update all affected spec files in the same
commit or PR. Piecemeal updates leave the corpus in a temporarily inconsistent state and
make it harder to verify propagation.
